Shape Specification Language Reference

Most framework services are driven by declarative linked data models defined using a SHACL‑based shape specification language.

Models are programmaticaly built using a Java‑based DSL and can eventually automate a range of different tasks in the lifecycle of linked data REST APIs.

task shape role
data selection RDF data retrieved from readable linked data REST APIs is selected from the underlying graph storage layer by SPARQL queries derived from the associated linked data models
data validation RDF data submitted to writable linked data REST APIs on resource creation and updating is automatically validated against the associated linked data models
faceted search faceted search and ancillary facet-populating queries are managed by the system on the basis of structural and typing constraints specified in linked data models associated with target linked data REST APIs
dynamic UI building Interactive linked data navigators and editing forms could be dynamically created and laid out according to the linked data models associated with linked data REST APIs
API documentation Human and machine readable docs for linked data REST APIs could be dynamically derived from associated linked data models and published as hypertexts and OpenAPI/Swagger specs
data ingestion RDF data ingested from external data sources could be dynamically mapped to and validated against the linked data models associated with target RDF graphs

The details of the specification language are still in flux, especially when it comes to SHACL interoperability.

Some of the described features aren't (yet ;-) supported by the framework.

# Shapes

Linked data shapes define the expected structure of RDF resource descriptions in terms of constraints imposed on members of a focus set of RDF terms and on the values of their RDF properties.

Primitive shapes specify declarative constraints to be meet by the focus set and its member terms. Composite shapes recursively assemble other shapes into tree-structured complex constraints.

Linked data selection tasks identify the set of subjects in an RDF graph whose RDF descriptions are consistent with a possibly composite shape. Selection results are reported either as an RDF graph or a structured report according to the choosen query type.

Linked data validation tasks verify that the RDF description of an initial focus set is consistent with the constraints specified by a possibly composite shape. During the process, derived focus sets connected by structural constraints may be recursively validated. Validation results are reported as a structured focus validation trace.

Annotations

Non-validating shapes documenting shape metadata, to be used for purposes such as shape redaction, form building or predictable encoding of RDF files.

shape value
meta(IRI, value, …) the metadata property identified by the given tag is associated with the given value in the enclosing shape
guard(axis, value, …) the focus set is consistent with this shape only if the value of an externally assigned axis variable is included in a given set of target values

Common metadata annotations are directly available as shorthand shapes.

shorthand shape value
alias("alias") an alternate property name for reporting values for the enclosing shape (e.g. in the context of JSON-based RDF serialization results)
label("label") a human-readable textual label for the enclosing shape
notes("notes") a human-readable textual description of the enclosing shape
placeholder("placeholder") a human-readable textual placeholder for the values of the enclosing shape
dflt(value) the default for the expected values of the enclosing shape
hint("IRI") the IRI of a resource hinting at possible values for the enclosing shape (e.g. an LDP container)
group("value") a client-specific group visualization mode for the enclosing shape
index(true|false) an indexing hint: true if the value of the enclosing shape should be indexed in the storage backed, false otherwise

Term Constraints

Primitive shapes specifying constraints to be individually met by each RDF term in the focus set.

shape constraint
datatype(IRI) each term in the focus set has a given extended RDF datatype IRI; IRI references and blank nodes are considered to be respectively of app:/terms#iri (Values.IRI) and app:/terms#bnode (Values.BNode) datatype
class(IRI) each term in the focus set is an instance of a given RDF class or one of its superclasses
minExclusive(value) each term in the focus set is strictly greater than a given minum value, according to SPARQL ordering rules
maxExclusive(value) each term in the focus set is strictly less than a given maximum value, according to SPARQL ordering rules
minInclusive(value) each term in the focus set is greater than or equal to a given minimum value, according to SPARQL ordering rules
maxInclusive(value) each term in the focus set is less than or equal to a given maximum value, according to SPARQL ordering rules
minLength(length) the length of the lexical representation of each term in the focus set is greater than or equal to the given minimum value
maxLength(length) the length of the lexical representation of each term in the focus set is less than or equal to the given maximum value
pattern("pattern") the lexical representation of each term in the focus set matches a given regular expression pattern
like("keywords") the lexical representation of each term in the focus set matches the given full-text keywords

Set Constraints

Primitive shapes specifying constraints to be collectively met by the RDF terms in the focus set.

shape constraint
minCount(count) the size of the focus set is greater than or equal to the given minimum value
maxCount(count) the size of the focus set is less than or equal to the given maximum value
in(value, …) the focus set is a subset a given set of target values
all(value, …) the focus set includes all values from a given set of target values
any(value, …) the focus set includes at least one value from a given set of target values

Common combinations of set constraints are directly available as shorthand shapes.

shorthand shape equivalent shape constraint
required and(minCount(1), maxCount(1)) exactly one
optional maxCount(1) at most one
repeatable minCount(1) at least one
multiple and() any number
only(value, …) and(all(value, …), in(value, …)) constant pre-defined values

Structural Constraints

Composite shapes specifying constraints to be met by a derived focus set generated by a path.

shape constraint
field(IRI, shape) the derived focus set generated by a single step path is consistent with a given shape

Inverse path steps may be generated using the Values.inverse(IRI) factory method.

Logical Constraints

Composite shapes specifying logical combinations of shapes.

shape constraint
and(shape, …) the focus set is consistent with all shapes in a given target set
or(shape, …) the focus set is consistent with at least one shape in a given target set
when(test,pass [, fail]) the focus set is consistent either with a pass shape, if consistent also with a test shape, or with a fail shape, otherwise; if omitted, the fail shape defaults to and(), that is it's always meet

Test shapes for conditional constraints are currently limited to combinations of parametric guards and logical operators.

# Parameters

The combined use of conditional (when) and parametric (guard) constraints supports the definition of parametric shapes, which specify different sets of constraints according to the current externally-assigned value of parametric axis variables.

Parametric axes may be specified for arbitrary custom variables, but the system relies on four pre‑defined parametric variables to support fine‑grained access control rules and role‑dependent read/write resource views.

Common combinations of parametric shapes on these axes are directly available as shorthand shapes.

Guards may be applied to groups of shapes either explicitely assembly a when shape, like: 

when(<guard>, and(<shape>, …))

or using the shorthand then method, like:

<guard>.then(<shape>, …)

Role Axis

Parametric guards for the role axis selectively enable target shapes according to the roles of the (possibly authenticated) user performing HTTP/S operations on target linked data resources.

shorthand usage context
role(value, …) target shapes are to be considered only if the sets of roles enabled for the user performing a request contains at least one of the given role values

Task Axis

Parametric guards for the task axis selectively enable target shapes according to the method of the HTTP/S operations performed on target linked data resources.

shorthand HTTP/S method usage context
create() POST resource creation
relate() GET resource retrieval
update([) PUT resource updating
delete() DELETE resouce deletion
client() POST+GET+DELETE shorthand for client-managed data, specified at creation time, but not updated afterwards
server() GET+DELETE shorthand for server-managed data, neither specified at creation time, nor updated afterwards

Area Axis

Parametric guards for the area axis selectively enable target shapes according to the usage context.

shorthand usage context
holder() collection holder description
member() collection member description (shorthand for digest+detail)
digest() short resource description, e.g. inside search result sets
detail() full resource description

Mode Axis

Parametric guards for the mode axis selectively enable target shapes according to the usage pattern.

shorthand usage pattern
convey() target shapes are to be used only for validating incoming data or extracting outgoing data and not for selecting existing resources to be processed
filter() target shapes are to be used only for selecting existing resources to be processed and not for validating incoming data or extracting outgoing data
